# Landing (Google Ads-ready) — НЕ менять контент, цвета и структуру

## Цель
Доработать текущий лендинг так, чтобы он был максимально “чистый” для Google Ads (прозрачность, политика, контакты, понятный user-flow),
при этом **сохранить существующий контент, цвета и структуру** (включая формулировки, заголовки, порядок блоков, CTA, визуальную стилистику).

## Критически важно (жёсткие ограничения)
### 1) НЕ МЕНЯТЬ контент
- Нельзя переписывать тексты, заголовки, подписи, CTA, вопросы/ответы квиза.
- Нельзя менять смысл, тон, обещания, цифры, формулировки.
- Нельзя добавлять “маркетинговые” утверждения в существующие блоки.

✅ Разрешено:
- Добавлять **новые** блоки/страницы (Privacy / Terms / Contacts / Disclaimer) внизу/в футере.
- Добавлять короткие пояснения **только в новых блоках**, не вмешиваясь в существующие.

### 2) НЕ МЕНЯТЬ дизайн (цвета/шрифты/структуру)
- Нельзя менять палитру, фон, кнопки, отступы, сетку, порядок секций.
- Нельзя менять размер шрифтов и визуальную иерархию.
- Нельзя “улучшать UX” перестановками и редизайном.

✅ Разрешено:
- Добавить **футер** (или мини-панель в самом низу страницы) в стиле сайта.
- Добавить ссылки на страницы: Privacy Policy / Terms / Contacts.
- Добавить маленький “disclaimer” внизу страницы мелким текстом (в едином стиле).

### 3) НЕ ДОБАВЛЯТЬ запрещённые элементы
- Никаких попапов с автопереадресацией.
- Никаких автоматических редиректов в Telegram без явного клика пользователя.
- Никаких скрытых загрузок/скриптов, которые меняют поведение страницы для модерации.

## Что нужно сделать (задачи)
### A) Структура проекта (предпочтительно)
- `index.html` — основной лендинг (текущий контент сохраняем как есть).
- `assets/` — css/js/img (если уже иначе — не ломать, но держать порядок).
- `privacy.html` — политика конфиденциальности
- `terms.html` — условия использования
- `contacts.html` — контакты / реквизиты (если есть)
- (опционально) `disclaimer.html` — отдельная страница дисклеймеров, если не помещается в футер.

> Важно: если проект сейчас на одном файле — можно оставить один файл, но всё равно добавить отдельные страницы или якорные секции внизу.
> Главное — НЕ менять текущие блоки.

### B) Добавить “обязательные” элементы для доверия/прозрачности (не трогая основной контент)
1) **Футер** в самом низу страницы:
   - Links: “Privacy Policy”, “Terms”, “Contacts”
   - Короткая строка копирайта: “© <год> <название/бренд>”
2) **Мини-дисклеймер** (в футере или сразу над ним, мелким текстом):
   - Что это “оценка правдоподобия речи/контента”, не фактчекинг.
   - Что пользователь сам принимает решения.
   - Что анализ в Telegram — следующий шаг (если так и есть), и это происходит **только по клику**.
3) Страницы:
   - `privacy.html`: какие данные собираем, зачем, хранение, третьи лица, cookies, аналитика, как удалить данные/запрос.
   - `terms.html`: правила использования, ограничения ответственности, возрастные ограничения (если нужно), запрет незаконного использования.
   - `contacts.html`: email для связи + (если есть) юр.инфо/страна/город. Если юр.данных нет — хотя бы рабочий email и форма/telegram-contact.

### C) Технические требования
- Страница должна открываться без ошибок.
- Никаких broken links.
- Mobile-first: всё корректно на телефоне.
- Скорость: не подключать тяжёлые библиотеки “просто так”.
- Не блокировать ботов (не ставить `noindex,nofollow`, не закрывать CDN/статические файлы).
- HTTPS обязателен.
- Если есть трекинги (Pixel/GA/GTM) — подключать аккуратно, без ошибок в консоли.

## Контент для новых страниц (шаблонно, но без самодеятельности)
### Privacy Policy — обязательно включить разделы
- Какие данные пользователь вводит (текст/аудио/контакт/технические).
- Цель обработки (оказание услуги/аналитика/улучшение сервиса).
- Правовое основание (consent/legitimate interest — кратко).
- Срок хранения (конкретно или “до X дней/до запроса удаления”).
- Передача третьим лицам (например, поставщики хостинга/аналитики/платёжки; если LLM API — описать как обработчика).
- Cookies/analytics.
- Как запросить удаление данных (email).
- Контакты оператора данных (email).

### Terms — обязательно включить
- Описание услуги как “информационной/развлекательной” (если так задумано).
- Запрет на незаконное использование.
- Ограничение ответственности.
- Право изменять сервис.
- Возраст/дееспособность (при необходимости).
- Контакты.

- (Опционально) страна/город

## Security / Проверка на взломанный сайт (обязательно)
Цель: исключить статус "Hacked site" / "Malware" / "Unwanted software" и любые причины отклонения Google Ads по безопасности.

##Требования:
- Никаких скрытых редиректов, клоакинга, подмены контента.
- Никаких подозрительных внешних скриптов/инъекций.
- Удалить любые неиспользуемые/подозрительные JS-файлы и подключения.
- Проверить, что на сайте нет:
  - iframe-инъекций
  - скрытых ссылок (display:none / offscreen)
  - странных inline-скриптов, obfuscation, eval(), base64-decode цепочек
  - неизвестных доменов в подключениях (script/src, img/src, link/href, fetch/xhr)

##Чеклист проверки перед релизом:
- [ ] Google Search Console → Security issues: нет предупреждений.
- [ ] Google Safe Browsing: домен не помечен как опасный.
- [ ] VirusTotal / Sucuri SiteCheck: чисто (по возможности).
- [ ] В DevTools → Network нет запросов к неизвестным доменам.
- [ ] В исходнике страницы нет obfuscated JS / eval() / подозрительных вставок.
